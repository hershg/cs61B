Linked List Data Structure with Java

  • Last time we created an IntList class that is basically a Java equivalent of the linked list classes we used in Python in CS 61A
  • IntList objects have a first attribute that holds some integer, and a rest that is itself an IntList object pointing to the rest of the linked list. In the case that this is the final/only link in the list, the rest is null

  • IntList class has the two instance variables followed by a constructor that assigns the first and rest input to there instance variables appropriately. Class has further methods later to give functionality (such as size/append/search methods)

    public class IntList {

      public int first;

      public IntList rest;

    

      public IntList(int f, IntList r) {

          first = f;

          rest = r;

      }

    }

Single Linked List (SLList): Improvement on IntList

  • This implementation is 'dangerous' because the function's recursion is visible publicly. We can improve this.
  • Lets start with a simple version of IntList called IntNode that doesn't have any extra functionality methods
  • We can define this class privately inside a superclass SLList. We also give SLList two instance variables sentinel (IntNode object storing the list data) and size (integer), and two constructors (polymorphism, one initializes to null, other to given integer)
  • We also give this class some functionality methods to make it more usable (addFirst,getFirst,addLast, get, and size)
  • Lastly, we have a main method that creates an SSList and uses it:
/** An SLList is a list of integers. It hides the ugly inner workings. */
public class SLList {
private static class IntNode {
public int item;
public IntNode next;
public IntNode(int i, IntNode n) {
item = i;
next = n;
}
}
/* The first item (if it exists) is at sentinel.next. */
private IntNode sentinel;
private int size;
/** Creates an empty SLList.
* The constructors start off with a extra sentinel node (-1)
* so that when initializing SSList's with out passing parameters,
* we don't have an error (without this, we would need to assign
* null to an integer which we can't do). In this class, we will
* access and edit SLLists starting from sentiinel.next, ignoring
* the existence of this sentinel node. */
public SLList() {
sentinel = new IntNode(-1, null);
size = 0;
}
public SLList(int x) {
sentinel = new IntNode(-1, null);
sentinel.next = new IntNode(x, null);
size = 1;
}
/** Adds x to the front of the list. */
public void addFirst(int x) {
sentinel.next = new IntNode(x, sentinel.next);
size = size + 1;
}
/** Returns the first item in the list. */
public int getFirst() {
return sentinel.next.item;
}
/** Adds x to the end of the list. */
public void addLast(int x) {
size = size + 1;
IntNode p = sentinel;
/* Advance p to the end of the list. */
while (p.next != null) {
p = p.next;
}
p.next = new IntNode(x, null);
}
/** Returns the ith item of this list. */
public int get(int i) {
IntNode p = sentinel;
/* Advance p to the ith position in list. */
while (i > 0) {
p = p.next;
i -= 1;
}
return p.next.item;
}
/** Returns the size of the list. */
public int size() {
return size;
}
public static void main(String[] args) {
/* Creates a list of one integer, namely 10 */
SLList L = new SLList();
L.addLast(1);
L.addLast(2);
L.addLast(3);
System.out.println(L.size());
int f = L.get(1);
}
}

SLList vs IntList:

  • To initialize list, with IntList we would need to pass in a null parameter. We improve the class by hiding the null parameter.
  • To add links to list, we can nicely call SLList methods, whereas before we had to deal with actually creating new instances of IntList class. By simplifying the interface for using the SLList class, we have improved it.
  • SLList class acts as 'middle man' between user and raw data structure
  • SLList has privatized its API (Application Protocol Interface) members, so it's more secure:
    • We can't modify/mess up where the pointers in the links point to from outside the class instance, so we can only perform safe, expected actions on our list structure which is good.
    • This also makes it safer to change the implementation of the internal private members
    • Less internal details for class users to understand
    • Doesn't really protect from hackers/spies/..., more for organization purposes
  • SSList has a private internal class IntNode. This is good because other classes shouldn't be manipulating the IntNode instance, and the IntNode instances don't stand on their own - they depend on a SLList superclass. Thus, make them nested and private.
  • The nested IntNode class doesn't use any of its superclass's instance variables, so we can declare it static
    • Remember, static classes can't access outer class's instance members
    • We can do this to IntNode, since it doesn't use the instance variables, constructors, or methods of SLList
    • Gives minor memory savings (8 bits)
  • SSList uses an integer variable to cache the size of the list, improving size lookup from linear to constant time.
  • In SSList, we made the initial node (the negative first node) a sentinel node to simplify adding functionality for instantiating and then adding to null SLLists (we generalized SSList).
  • Invariants: conditions that are guaranteed to be true
    • For example, we know the first actual node is at sentinel.next, and that sentinel always refers to the sentinel node preceding the actual list data
    • We assume the invariants to make it easier to simplify/modify/edit code (ex: so methods like addLast don't have to worry about nulls), but we must make sure all methods preserve the invariants.

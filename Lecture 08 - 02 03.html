Lecture 08 - 02/03

Inheritance, Subtype Polymorphism

  • The AList and SLList classes have the exact same core methods (get, addFirst, addLast, size, ...)
  • If we had some method that manipulates lists, and we wanted it to be able to take both an SLList type list and an AList type list, we could have the exact same function twice with the type for the list structure parameter switched. This idea of the same named method taking different parameters of different types is called overloading
  • However, this is ugly since we are reusing almost the exact same code twice. Furthermore, it won't work for future list structure types, and is hard to maintain (bug fixed in AList doesn't mean bug fixed in SLList)

Hypernyms and Hyponyms

  • Hypernyms are words that group/classify a set of objects; these classified objects are hyponyms
  • Ex: Dog is a hypernym of poodle and golden retriever; poodle and golden retriever are hyponyms
  • We can use this hypernym/hyponym hierarchy in our programs with inheritance and interfaces

Interfaces:

  • Specifications of what a class should do, but doesn't have implementation
  • For example, we can have a List61B interface (hypernym), and have AList and SLList implement the interface (hyponyms)
public interface List61B<Item> {
void addFirst(Item x);
void addLast(Item y);
Item getFirst();
Item getLast();
Item removeLast();
Item get(int i);
void insert(Item x, int position);
int size();
default void print() {
for (int i = 0; i < size(); i = i + 1) {
System.out.print(get(i) + " ");
}
}
}
  • Now, we make the other classes implement this interface:
public class AList<Item> implements List61B<Item>{

  ...

  @Override

  public void addLast(Item x) { 

    ...

  ...
  • We can now adjust our original method that manipulates/uses a list type data structure, by changing the parameter from a list of type AList or SLList instead to a parameter of type List61B
  • Note that we used @Override annotation. This isn't necessarily needed but gives the compiler an additional thing to check and it's a very good coding practice. We will be using this in this course.
    • If a subclass method has the same signature as a superclass method, the subclass method overrides the superclass method, regardless of the annotation. The annotation just reminds that we are overriding
    • This will protect against typos. For example, if we give some method an override annotation but it isn't actually overriding a method from its superclass (most likely due to a typo), then we will get a compile error and catch this
  • The usage of the implements keyword is known as interface inheritance

  • We can also do things like the following:

    public static void peek(List61B<String> list) {

      System.out.println(list.getLast());

    }

    public static String peek(SLList<String> list) {

      System.out.println(list.getFirst());

    } 

    SLList<String> SP = new SLList<String>();

    List61B<String> LP = SP;

    ...

    peek(SP);

    peek(LP);

    Note on line 8, we assign a variable of type interface to a variable of type subclass. This is valid. The result of this is line 11 will use List61B's peek method instead of SLList's method (line 10 uses SLList's peek method)

  • Java also allows implementation inheritance

    • Subclasses inherit not just the signatures, but also implementations
    • Interface methods with this specify not just what a List61B should do, but also how
    • Done with the default keyword
    • Example: the print method in the above interface for List61B
    • Can still be overridden by subclasses

Dynamic Method Selection:

Static Type vs. Dynamic Type

  • Every variable in Java has a “compile-time type”, a.k.a. “static type”
    • This is the type specified at declaration. Never changes
  • Variables also have a “run-time type”, a.k.a. “dynamic type”
    • This is the type specified at instantiation (e.g. when using new)
    • Equal to the type of the object being pointed at
  • For example, compile-time type can be an interface (ex: List61B), whereas run-time type can be a subclass that implements the interface (ex: SLList)
  • If the run-time type has a method that overrides the compile-time type's method, the run-time type's method is used (so SLList overrides are used over List61B interface methods). Known as Dynamic Method Selection
  • Dynamic method selection happens only for overridden methods (subclass overriding superclass), not for overloaded methods (when some class has two methods, one for a supertype and one for subtype)

Inheritance II

  • When a class is a hyponym of an interface, we used implements
  • If you want one class to be a hyponym of another class, you use extends
  • With extends, a subclass inherits the following from its superclass:
    • All instance and static variables
    • All methods
    • All nested classes
    • Constructors not inherited
  • However, note that some members can be private and thus inaccessible
  • Still can use @Override to change subclass's implementation of superclass's method
  • Constructors: a subclass's constructor must start with a call to the superclass's constructor
    • Call superclass's constructor with super();
    • If we don't explicitly call the superclass constructor, Java automatically does this (but for example may not pass appropriate parameters into the construtor)
  • All Java types extend from Object class
  • At runtime, the this for a call to a subclass is the subclass instance itself (remember run-time/dynamic type)
    • Let both the subclass and superclass have methods of the same name foo
    • Let the superclass have a method bar that calls foo
    • Let the subclass not have a method named bar
    • If at some point, some method of the subclass calls bar, since the subclass doesn't have the method, its superclass's method of the same name, bar, is called
    • This bar method in turn call the foo method of the subclass, since this refers to the subclass
    • Note: this can lead to abstraction problems if the workings of a superclass are not known (thus it is said that implementation inheritance can break the abstraction barrier/encapsulation)

Encapsulation

  • A problem with building large programs is complexity
  • We counter complexity with hierarchical abstraction - creation of layers of abstraction, abstraction barriers
  • Improve programs by designing them for change
    • Organize programs around objects
    • Hide information others don't need
  • Module: A set of methods that work together as a whole to perform some task or set of related tasks
  • A module is said to be encapsulated if its implementation is completely hidden, and it can be accessed only through a documented interface
    • Privatize instance variables and helper methods (helper functions not for outside use)
  • Encapsulated modules have an abstraction barrier, so as users we can't see internals (also shouldn't usually look at internals when writing tests)
  • Implementation inheritance (such as extends) breaks encapsulation

Type Checking/Casting

  • Compiler allows method calls/variable assignments based on the static/compile-time type of a variable
  • Example of this, along with static/comile-time vs dynamic/run-time types:
    • Say we have a superclass superC and a subclass subC that extends superC
    • Let our subclass has a method foo that the superclass doesn't
    • Let superC and subC both have a method bar that subC overrides, and let the superclass have a method baz that the subclass lacks
    • Then, from main, we can say subC mySub = new subC; followed by superC mySuper = mySub; (this is allowed since a subclass is of the type of its superclass)
      • Note that the compile-time/static type of mySuper will be superC but the run-time/dynamic type is subC
    • Continuing from main, if we do mySuper.baz();, we will call the baz method of the superclass since the subclass inherits this method
    • Then, if we do mySuper.bar();, we will use that of the subclass since we call based on the dynamic type of mySuper, which is subC
    • Finally, if we try to do mySuper.foo();, we will actually get a compile-time error, and not call the subclass method foo
      • This is because the Java compiler allows method calls based on the compile-time type, which is superC. Note, we didn't have this problem with the bar method because even though we end up calling the subclass method, the compiler sees that the compile-time type of superC does indeed have such a class and allows this
      • Basically, the compiler doesn't know what mySuper's run-time type will be, and doesn't want to allow such a call since it doesn't know what will happen (compiler very conservative)
    • The compiler does a similar check with variable assignments
      • So, for example, we would get a compile-time error if we did subC mySub2 = superC; since the Java compiler doesn't know the run-time type of superC and plays it safe
  • The new keyword specifies the compile-time type of an expression (ex: for the part of an assignment that declares new someClass, this part of the assignment has a compile-time type of someClass)
  • Type casting: to force the compile-time type of an expression, we put the desired type in parenthesis before the expression (ex: (thisClass) mySomeClass tells the compiler that mySomeClass has a type thisClass, and thus is treated as such (of course, this could lead to a run-time error if this type casting is invalid))

Higher Order Functions

  • These exist in Java as well as Python
    • In Java 7 and older, we couldn't have a variable be a pointer to a function, making HOF tricky; we can make HOFs with interfaces instead to overcome this
    • Don't have this problem in Java 8
  • A HOF is a function that treats another function as input

For the following in Python:

def tenX(x):

    return 10*x



def do_twice(f, x):

    return f(f(x))



print(do_twice(tenX, 2))

We can equivalently do the following in Java via interfaces:

public interface IntUnaryFunction {

    int apply(int x);

}



public class TenX implements IntUnaryFunction {

    public int apply(int x) {

           return 10 * x;

    }

}



public class HoFDemo {

    public static int do_twice(IntUnaryFunction f, int x) {

           return f.apply(f.apply(x));

    }

    public static void main(String[] args) {

           System.out.println(do_twice(new TenX(), 2));

    }

}

Summary:

  • For inheritance with the extends keyword, a subclass gets the following non-private members from a superclass
    • Variables, methods, nested classes
    • Not constructors
  • Subclass constructor must invoke superclass constructor first
  • Use super to invoke overridden superclass methods and constructors
  • Invocation of overridden methods follows these rules:
    • Compiler plays it safe and only lets us do things allowed by static type
    • For overridden methods, the actual method invoked is based on dynamic type of invoking expression
      • This is bad to do for overloaded methods
  • Can use casting to overrule compiler type checking

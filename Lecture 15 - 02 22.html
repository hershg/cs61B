Packages, Access Control, Objects

Packages, JAR files, Build Systems

  • Package is namespace organizing classes and interfaces, for example the edu.berkeley.cs.61B package (full expression is canonical name, the last part 61B is simple name)
  • Package name placed at top of all files in package
  • Package folder storage must match package name (edu.berkeley.cs.61B in edu/berkeley/cs/61B)
  • To use a class from package A in package B, we must use the full canonical name
    • Let there be a class 61BList in edu.berkeley.cs.61B package
    • Then we can use the canonical name edu.berkeley.cs.61B.61BList obj = new ...
    • Alternatively, we can import edu.berkeley.cs.61B and then use the simple name 61B obj = new ...
  • All Java classes without package name at top of file are part of the default package (should avoid putting code here for moderate-large sized programs)
    • Cannot import from the default package
  • Can share Java code by creating a JAR (.jar) file that contains the relavant files and necessary dependencies (these are essentially zip files; not secure, just for sharing/transporting code)
  • Build Systems like Ant, Maven, and Gradle are used for large projects with large teams. These are very useful, and Gradle is getting really popular

Access Control

  • Let there exist an object X declared in location L, with modifier M (public/protected/package private/private)
  • This table tells us whether we can access X if we are in the same class/same package/subclass of L/anywhere, depending on X's modifier M

Access is based only on static types:
Let us have the following code:

package universe;

public interface BlackHole {

    void add(Object x);

}

Note that add method is actually of public access, as it inherits the modifier of the interface it's defined in because it lacks a modifier. In cases like this, the lack of a modifier doesn't indicate packageprivate (Java's weird like that)

package universe;

public class CreationUtils {

    public static BlackHole hirsute() {

        return new HasHair();

    }

}

package universe;

class HasHair implements BlackHole {

    Object[] items;

    public void add(Object o) { ... }

    public Object get(int k) { ... }

}

And lastly the file below (note this is in the default package, not universe package):

import static CreationUtils.hirsute;

class Client {

    void demoAccess() {

        BlackHole b = hirsute();

        b.add("horse");

        b.get(0);

        HasHair hb = (HasHair) b;

    }

}
  • Line 4 above will compile since from the import statement, we do indeed have a hirsute method that is of public access
    • Note that b has static type Blackhole and dynamic type HasHair
  • Line 5 will also compile, since we look at b's static type Blackhole and see that it indeed has a public method add
  • Line 6 will not compile, since the static type of b doesn't contain a get method
  • Line 7 will also not compile, since we actually can't even see the HasHair class since we are outside of it's package
  • Note: we cant refer to a HasHair object from outside the package, but can still call methods that have dynamic type HasHair from outside the package, as demonstrated in this example

Access control at top level (on entire interface/class):

  • We can declare entire classes as public or no modifier/package private
  • Only these two levels (no private/protected class, and no such thing as sub-packages - any difference in canonical package name means completely different package)
  • These modifiers apply to all the members of the class/interface

Access control purpose:

  • Private declarations are parts of the implementation of a class that only that class needs.
  • Package private declarations are parts of the implementation of a package that other members of the package will need to complete the implementation.
  • Protected declarations are things that subtypes might need, but subtype clients will not.
  • Public declarations are declarations of the specification for the package, i.e. what clients of the package can rely on. Once deployed, these should not change.

Objects

  • All classes are hyponyms of the Object class (which is not an abstract class)
  • As a result, they inherit several methods, including .toString and .equals
  • We can override these in a class if we want to change method behavior

  • .equals calls the appropriate function to compare two classes, to see if they are equal to each other

    • == just compares the bits the two variables reference, to see if those point to the same object (comparing bits)
    • Default implementation of .equals uses ==
    • For arrays, we use Arrays.equal or Arrays.deepEquals
    • .equals methods must be reflexive, symmetric, and transitive
    • They must take an Object argument
    • They must always return false for a null argument

    A common equals implementation:
    Let us have a class Foo with three member variables. The following could be it's equality method:

    @Override

    public boolean equals(Object x) {

      if (this == x) return true;

      if (x == null) return false;

      if (this.getClass() != x.getClass()) {

        return false;

      }

      Foo that = (Foo) x;

      if (this.memVar1 != that.memVar1) {

        return false;

      }

      if (this.memVar2 != that.memVar2) {

        return false;

      }

      if (this.memVar3 != that.memVar3) {

        return false;

      }

      return true;

    }
